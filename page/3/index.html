The View Controller Hierarchy 视图控制器的层次结构

By Wen Yongyang

发表于 2015-11-29

应用中的 view controllers 之间的关系定义了一系列每个 view controller 必要的行为方法。UIKit 希望开发者按照预定的思路来使用 view controllers。维持 view 正确控制的关联能够保证自动化的行为能够传递到正确的 view controllers 中。如果开发者不遵守原有的控制逻辑与呈现之间的关联，那么应用的部分功能将不会得到预期中的效果。

The Root View Controller 根控制器

根控制器是整个视图控制器层次结构的基础。每一个窗口(window)都只有一个根控制器，而且这个控制器的内容填充整个屏幕。根控制器定义了所有用户能够看见的基础内容。Figure 2-1 展示了根控制器与其窗口的关系。因为窗口没有它自己的可视化内容，所以控制器的视图提供了所有要展示的内容。

根控制器可以通过 UIWindow 对象的 rootViewController 属性访问得到。当你使用 storyboards 来设置你的控制器时，UIKits 会在应用启动阶段自动设置该属性。当你使用代码来创建 window 对象时，你必需亲自设置根控制器。

Container View Controllers 容器类视图控制器

容器类视图控制器能让你使用可管理和可充用的部件来构建复杂的交互界面。容器类视图控制器融合了一个或多个拥有可定制化的视图的子视图控制器来创建它最终的界面。例如：UINavigationController 对象整合呈现了一个子视图控制器，一个导航栏和一个可选的工具栏，这些控件，都是通过导航栏控制器（navigation controller）来进行管理。UIKit 包含了多个容器类视图控制器，包括：UINavigationController, UISplitViewController, UIPageViewController。

容器类视图控制器的视图，总是会填满提供给它的所有空间。容器类视图控制器通常会设置为窗口的根控制器（如 Figure 2-2 所示），但它们也可以模态显示或者作为其它容器的子控制器。容器控制器对设置它的子视图位置负责。在 Figure 中，容器控制器将两个子视图靠边摆放。尽管子视图的位置由容器的大小决定，但子视图控制器可以以最低限度来获知其它容器和其它同级的视图控制器。

由于容器视图控制器管理着它的子视图，所以 UIKit 定义了开发者如何在自定义的容器中设置容器的子视图控制器的规则。
见 Implementing a Container View Controller

Presented View Controllers 呈现视图控制器

呈现一个视图控制器将会取代正在呈现的视图控制器的内容，通常是隐藏当前的视图控制器的内容。Presentations 经常用于模态显示新的内容。例如：你可以呈现一个视图控制器来收集用户的输入。你也可以使用使用它们来作为应用交互界面基本构成。

当你呈现一个视图控制器时，UIKit 会创建从正在呈现的视图控制器（presenting view controler）到之前呈现的视图控制器（presented view controller）之间的关系，如 Figure 2-3 所示（也有从之前呈现的视图控制器到正在呈现的视图控制器间的关系）。这些关系形成了视图控制器层次结构。同时，这些关系用于在运行时定位其它视图控制器。

当使用到容器视图控制器时，UIKit 可以修改视图呈现链（presentation chain）来简化开发者需要编写的代码。不同的呈现风格由不同的呈现规则。例如：全凭的呈现方式总是覆盖整个屏幕。当你呈现一个视图控制器时，UIKit 会寻求一个视图控制器，该视图控制器提供一个合适的上下文环境来呈现。大多数情况下，UIKit 会选择最近的容器视图控制器，但它也可以选择窗口的根控制器。而其它情况下，你也可以告知 UIKit 使用哪个视图控制器来定义呈现的上下文和呈现的方式。

Figure 2-4 展示了为什么容器经常为视图呈现提供上下文。当进行一个全屏的视图呈现时，新的视图控制器需要覆盖整个屏幕。这种情况下，是容器决定是否处理视图的呈现，而不是让子视图获知容器的尺寸大小。因为在这个例子中，导航栏控制器覆盖整个屏幕，因此，它就运行得像要呈现的视图一样，并且负责初始化视图的呈现。

The Role of View Controllers 视图管理器的作用

By Wen Yongyang

发表于 2015-11-29

View Controller 视图控制器

View Controllers 是应用内部结构的基础。每一个应用至少有一个 View Controllers，但大部分的应用会有多个。每一个 View Controllers 管理应用中部分的用户交互界面以及用户交互界面与低层数据的交互。同时，使用 View Controllers 也有利于应用中不同部分界面之间的相互过渡。

View Controllers 在应用中扮演着非常重要的角色，因此，View Controllers 几乎是你工作的重心。UIViewController 类定义了多个方法(methods) 和属性(properties) 来管理视图(views)，事件处理(handling events)，视图控制器间的过渡切换(transitioning from one view controller to another)，协调处理应用的各部分。开发者应该继承 UIViewController （或其子类），并在这些类中添加自定义的代码来实现应用的功能。

其中，有两种类型的 View Controllers:

  • Content view controllers（内容类视图控制器），管理应用內零散的内容，这是开发者主要会创建的 view controller
  • Container view controller（容器类视图管理器），从其他 view controllers（child view controllers）收集信息，并以导航的方式或以其他方式将其他的 view controllers 展现出来。

大多数应用都是上述两类 view controller 的混合体。

##View Management 视图管理

View controller 最重要的功能就是管理视图之间的层次关系。每一个 view controller 都有唯一的一个 root view（根视图），该 root view 包含了 view controller 的所有内容。在这个 root view 上，开发者添加想要展现的内容。Figure 1-1 描述了 view controller 和它的 views 之间的内部关系。View controller 总是有对其 root view 的引用，而每一个 view 都有对其 subviews 的强引用。

Figure 1-1Relationship between a view controller and its views

NOTE
使用 outlet 访问其他在 view controller 的 view 层次结构中的 view（如子控件）是一种很通用的做法。由于 view controller 管理其所有 view 的内容，因此，outlet 为开发者存储了对 views 进行访问时需要的引用。当 views 从 storyboard 加载时，outlet 会自动与 view 的实例关联起来。

Content view controller 只管理自身包含的 views。Container view controller 管理自身包含的 views 以及它的一个或多个 child view controllers 中的 root views。Container 并不管理其子控制器的 content。它只管理 root view，根据 container 的设计来计算大小位置并进行放置。Figure 1-2 描述了 split view controller 和它的子控制器。Split view controller 管理其 child views 的总体大小和位置，但 child view controller 才是真正管理那些 views 的控制器。

Figure 1-2 View controllers can manage content from other view controllers

##Data Marshaling 数据整理

View controller 扮演一个中间人(intermediary)的角色，协调它管理的 views 和应用的数据。UIViewController 类中的方法和属性能够让开发者管理应用的界面呈现效果。当继承 UIViewController 后，开发者可以向这个子类添加管理数据所需要的变量，向这个子类添加自定义变量时，会创建如 Figure 1-3 所示的关联，在这里，view controller 对数据和呈现数据的 views 都会有 references（引用）。在 view controller 和 view 之间来回传递数据是开发者的职责。

Figure 1-3A view controller mediates between data objects and views

开发者应明确清晰地对 view controllers 和数据对象有一种清晰的任务分工。大多数确保数据结构统一正确的逻辑工作应该被包含在数据对象自身中(即大多数数据处理逻辑代码应该写在数据类中)。而 view controller 则可以对从 views 来的数据进行验证并将数据打包成数据对象需要的特定格式，但要记住的是，要尽可能降低 view controller 在管理数据的任务。

使用 UIDocument 对象是独立于 view controllers 外管理数据的一种方法。Document 对象是一个能如何进行持久化存储的 controller 对象。当继承这个类时，你可以在其中添加你需要的方法或逻辑来读取数据或将数据传递到其他控制器或其他部分。View controller 可以存储它接收到的数据的一份副本，以此来方便更新视图，但 document 对象仍然拥有着真正的数据。

##User Interactions 用户交互

View controllers 是响应事件的对象，并具有处理从 responder chain（响应链） 传递下来的事件的能力。尽管如此，view controllers 很少直接处理触摸事件。相反，views 通常会处理它们自身的触摸事件，并将结果返回到一个 associated delegate（关联的代理方法）或一个 target object（目标对象，通常为当前的 view controller）。因此，大多数在 view controller 中的事件会通过 delegate methods（代理方法）或 action methods（动作方法）来处理。

More Info: Event Handling User Interactions And Event Handling Guide for iOS

##Resource Management 资源管理

View controller 默认对所有其创建的 views 和对象负责。UIViewController 类自动处理大多数层面的视图管理。例如：UIKit 自动释放那些不再需要的与视图关联的资源。在开发者自己创建的 UIViewController 子类中，开发者承担管理所有自己显式创建对象的责任。

当可用内存空间不足时，UIKit 请求应用释放那些不再需要的资源。其中一个请求方法就是系统调用 view controllers 中的 didReceiveMemoryWarning 方法。开发者使用这个方法，在里面编写代码，移除那些不再需要或者可以稍后重新创建的资源的引用。例如：开发者可以使用这个方法来移除缓存数据。当内存空间不足时，开发者应释放尽可能多的内存空间。而消耗太多资源的应用可能会被系统终止并完全移出内存来恢复内存空间。

##Adaptivity 适配

View controller 负责呈现 views 和调整呈现方式来匹配低层环境的工作。每一个 iOS 应用都应该能运行在 iPad 和不同尺寸的 iPhone 上。简单的做法是使用单一的 view controller 来调整 views 来适配不同的空间变化，而不是为每一种设备编写不同的 view controllers 和 view 层次结构。

在 iOS 中，view controller 需要处理 coarse-grained 变化和 fine-grained 变化。Coarse-grained 变化是当一个 view controller 的 trait 发生变化时产生。 Traits 是描述整体环境的属性，例如 display scale（显示比例）。其中最重要的两个 traits 是 view controller 的水平和垂直方向的 size classes （尺寸比列），这两个 traits 表示了 view controller 在给定的维度中有多大的空间。开发者可以使用 size class 变化来改变 views 的布局方式，如 Figure 1-4 所示。当水平 size class 为 regular 时，view controller 利用额外的水平空间来布局它的内容。当水平 size class 为 compact 时，view controller 垂直布局它的内容。

Figure 1-4 Adapting views to size class changes

在一个给定的 size class 中，给了 fine-grained 大小发生变化的可能。当用户将 iPhone 从 portrait（竖屏）转换为 landscape（横屏）时，size class 也许不会发生变化，但屏幕的尺寸却会发生改变。当开发者使用 Auto Layout （自动布局）时，UIKit 会自动调整 views 的大小和位置来适配新的尺寸。View controllers 可根据需要添加额外的调整。

More Info: The Adaptive Model

Linux 中 profile, bashrc, bash.profile

By Wen Yongyang

发表于 2015-11-23

  • etc/profile

    • 为系统每个用户设置环境信息
    • 用户第一次登录时被执行
    • 并从 /etc/profile.d 目录的配置文件中搜集shell的设置
    • 需重启生效

  • /etc/bashrc

    • 为每一个运行 bash shell 用户执行此文件
    • bash shell 被打开时，该文件被读取

  • ~/.bash_profile

    • 每个用户使用该文件设置专用于自己的 shell 信息
    • 用户登录时，改文件仅执行一次
    • 需重启生效

  • ~/.bashrc

    • 含专用于你的 bash shell 的 bash 信息
    • 登录时以及每次打开新的 shell 时,该文件被读取
    • 每个用户都有一个 .bashrc1 文件，在用户目录下

  • ~/.bash_logout

    • 每次退出系统(退出bash shell)时,执行该文件

实现可伸缩的 tableViewCell

By Wen Yongyang

发表于 2015-11-22

大致描述：
点击其中一个 Cell，实现在该 Cell 下伸展出更大的视图，展示更多的细节

思路：

  1. 所谓可伸缩的 Cell，其实是 header 与 cell 共同配合的结果
  2. 点击 header，伸展出该 header 下的 cell
  3. 使用 UIButton 覆盖 header，并在点击 UIButton 时实现伸缩效果

当 UITableView 展示 cell 时，代理方法的调用顺序大致为：

  • - (NSInteger)tableView: numberOfRowsInSection: // 查询要展示多少个 row
  • - (CGFloat)tableView: heightForRowAtIndexPath: // 查询每个 row 的高度
  • - (UITableViewCell *)tableView: cellForRowAtIndexPath: // 查询每个 row 长什么样子
  • - (void)tableView: didSelectRowAtIndexPath: // 监听每个 Cell 的点击事件

可以推出，tableView 的每一个 header 也是按照相同的顺序

数据源问题：
每一个 Cell 长什么样子，展示什么内容，必定是从数据源中获取，通常为数组

然而，在此例子中，我们还需要记录下每个 Cell 展开/收缩的状态，也许，我们可以另外新建一个数组来记录 cell 的状态，
但，我们还可以使用 字典 这个东西，将 数组 与 字典 配合使用

  1. 整个数据源的基础，我们还是利用 数组 (NSMutableArray) 来实现，我们称它为 cellDatas
  2. 而在 cellDatas 中，每一个元素便是 字典 (NSMutableDictionary)
  3. 在这个字典中，我们添加的字段有：数据的信息 (objInfo)，cell 的展开状态 (expanded)

在 - (NSInteger)numbersOfSectionsInTableView: 中：
我们返回的是 cellDatas 的个数

在 - (CGFloat)tableView: heightForHeaderInSection: 中：
我们返回的是 header 的高度

在 - (UIView *)tableView: viewForHeaderInSection: 中：
我们返回的是 我们自定义的 header，即那个带有 UIButton 的View

在 - (NSInteger)tableView: numberOfRowsInSection: 中：
我们 根据 cellDatas 中，特定位置的 cellData 的 expanded 键值 (YES/NO) 来返回
如果返回的是 NO，即当前 cell 并没有展开，于是我们返回 我们需要的值，以便在 reloadData 时展开
如果返回的是 YES，即当前 cell 已经展开了，于是我们返回 0，以便在 reloadData 时收缩
为此，我们可以通过一个自定义方法 - (BOOL)isExpanded:(NSInteger)section 来判断，要注意的是，在该方法中，我们需要判断 cellData 中 expanded 键值是否为空，否则会崩溃

而关于 cell 的高度，以及展示样式，与 header 类似
要注意的时，当我们在 - (NSInteger)tableView: numberOfRowsInSection: 中返回 0 时，tableView 便不再访问其 高度，样式，因此，我们不需要在 该两个委托 中，检验是否展开了！

而后，
当我们点击 header 上的大按钮时，我们就要对 tableView 进行重新加载，展开/收缩 cell
为此，我们可以自定义一个方法 - (void)btnClicked，
在该方法中，我们需要对 cellData 的 expanded 键值进行设置，控制展开/收缩
为此，我们可以又自定义一个方法 - (void)collapseOrExpand:(NSIteger)section，
于是，在 - (void)btnClicked 中，我们首先 collapseOrExpand: 设置新的 cell 状态，然后 reloadData 根据 cell 的新状态重新加载 tableView，最后，可伸缩的 tableViewCell 就实现了

然而，问题又来了，在向上滑动 tableView 时，我们会看到每一个 section 的 header 浮动在最顶端
为了解决这个问题，我们需要实现 UIScrollView 的一个委托方法：

- (void)scrollViewDidScroll:(UIScrollView *)scrollView

更新，这里设置 tableView 的样式为 group

在这个委托方法中，我们的思路是：
判断条件A：scrollView.contentOffset.y <= header 的高度 && scrollView.contentOffset.y >= 0
判断条件B：若 A 不满足，则若 scrollView.contentOffset.y >= header 的高度

场景解析：
A：scrollView 向上滑动的距离小于或等于 header 高度这个长度，并且 scrollView 是在向上滑动
B：当 A 不满足时，我们滑动的距离已经大于了 header 高度这个长度，也许第二节的数据也已经滑出了显示界面

在 A 成立的情况下，scrollView.contentInset = [UIEdgeInsetsMake(-scrollView.contentOffset,y, 0, 0, 0)]
在 B 成立的情况下，scrollView.contentInset = [UIEdgeInsetsMake(-header 的高度, 0, 0, 0)]
