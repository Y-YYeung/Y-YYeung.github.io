实现可伸缩的 tableViewCell

By Wen Yongyang

发表于 2015-11-22

大致描述：
点击其中一个 Cell，实现在该 Cell 下伸展出更大的视图，展示更多的细节

思路：

  1. 所谓可伸缩的 Cell，其实是 header 与 cell 共同配合的结果
  2. 点击 header，伸展出该 header 下的 cell
  3. 使用 UIButton 覆盖 header，并在点击 UIButton 时实现伸缩效果

当 UITableView 展示 cell 时，代理方法的调用顺序大致为：

  • - (NSInteger)tableView: numberOfRowsInSection: // 查询要展示多少个 row
  • - (CGFloat)tableView: heightForRowAtIndexPath: // 查询每个 row 的高度
  • - (UITableViewCell *)tableView: cellForRowAtIndexPath: // 查询每个 row 长什么样子
  • - (void)tableView: didSelectRowAtIndexPath: // 监听每个 Cell 的点击事件

可以推出，tableView 的每一个 header 也是按照相同的顺序

数据源问题：
每一个 Cell 长什么样子，展示什么内容，必定是从数据源中获取，通常为数组

然而，在此例子中，我们还需要记录下每个 Cell 展开/收缩的状态，也许，我们可以另外新建一个数组来记录 cell 的状态，
但，我们还可以使用 字典 这个东西，将 数组 与 字典 配合使用

  1. 整个数据源的基础，我们还是利用 数组 (NSMutableArray) 来实现，我们称它为 cellDatas
  2. 而在 cellDatas 中，每一个元素便是 字典 (NSMutableDictionary)
  3. 在这个字典中，我们添加的字段有：数据的信息 (objInfo)，cell 的展开状态 (expanded)

在 - (NSInteger)numbersOfSectionsInTableView: 中：
我们返回的是 cellDatas 的个数

在 - (CGFloat)tableView: heightForHeaderInSection: 中：
我们返回的是 header 的高度

在 - (UIView *)tableView: viewForHeaderInSection: 中：
我们返回的是 我们自定义的 header，即那个带有 UIButton 的View

在 - (NSInteger)tableView: numberOfRowsInSection: 中：
我们 根据 cellDatas 中，特定位置的 cellData 的 expanded 键值 (YES/NO) 来返回
如果返回的是 NO，即当前 cell 并没有展开，于是我们返回 我们需要的值，以便在 reloadData 时展开
如果返回的是 YES，即当前 cell 已经展开了，于是我们返回 0，以便在 reloadData 时收缩
为此，我们可以通过一个自定义方法 - (BOOL)isExpanded:(NSInteger)section 来判断，要注意的是，在该方法中，我们需要判断 cellData 中 expanded 键值是否为空，否则会崩溃

而关于 cell 的高度，以及展示样式，与 header 类似
要注意的时，当我们在 - (NSInteger)tableView: numberOfRowsInSection: 中返回 0 时，tableView 便不再访问其 高度，样式，因此，我们不需要在 该两个委托 中，检验是否展开了！

而后，
当我们点击 header 上的大按钮时，我们就要对 tableView 进行重新加载，展开/收缩 cell
为此，我们可以自定义一个方法 - (void)btnClicked，
在该方法中，我们需要对 cellData 的 expanded 键值进行设置，控制展开/收缩
为此，我们可以又自定义一个方法 - (void)collapseOrExpand:(NSIteger)section，
于是，在 - (void)btnClicked 中，我们首先 collapseOrExpand: 设置新的 cell 状态，然后 reloadData 根据 cell 的新状态重新加载 tableView，最后，可伸缩的 tableViewCell 就实现了

然而，问题又来了，在向上滑动 tableView 时，我们会看到每一个 section 的 header 浮动在最顶端
为了解决这个问题，我们需要实现 UIScrollView 的一个委托方法：

- (void)scrollViewDidScroll:(UIScrollView *)scrollView

更新，这里设置 tableView 的样式为 group

在这个委托方法中，我们的思路是：
判断条件A：scrollView.contentOffset.y <= header 的高度 && scrollView.contentOffset.y >= 0
判断条件B：若 A 不满足，则若 scrollView.contentOffset.y >= header 的高度

场景解析：
A：scrollView 向上滑动的距离小于或等于 header 高度这个长度，并且 scrollView 是在向上滑动
B：当 A 不满足时，我们滑动的距离已经大于了 header 高度这个长度，也许第二节的数据也已经滑出了显示界面

在 A 成立的情况下，scrollView.contentInset = [UIEdgeInsetsMake(-scrollView.contentOffset,y, 0, 0, 0)]
在 B 成立的情况下，scrollView.contentInset = [UIEdgeInsetsMake(-header 的高度, 0, 0, 0)]

之后，问题又来了：
加载数据的圈圈只能在滑动的时候显示出来

By Wen Yongyang

假定有 B 回传值给 A

  1. 在 B.h 中使用 typedef 定义一个 Block
  2. 在 B.h 中定义一个一个方法，该方法使用上面定义的 Block 作为参数
  3. 在 B.m 中定义一个属性，该属性就是上面定义的 Block
  4. 在 B.m 实现 B.h 定义的方法，在方法中，将传入的参数 赋值 给 Block 属性
  5. 在 B 消失之际，调用 Block 属性，调用前需检查是否为空
  6. 在 A.m 中，实例化 B，并进行跳转前，通过 B.h 中暴露的方法，将要执行的代码作为参数传入 Block

消息传递

By Wen Yongyang

发表于 2015-11-22

使用 NSNotification

观察者模式

注册，成为观察者，并在其中实现收到消息的处理方法
[[NSNotificationCenter defaultCenter] addObserver:self selector:@selector(notice:) name:infoName object:nil];

addObserever:self 使自己成为观察者
selector:@selector() 收到消息的处理方法
name: 消息的专有名字
object: 消息的发送者，只接受此发送者发送的消息，传 nil 接受任何发送者

发送消息：
[[NSNotificationCenter defaultCenter] postNotificationName:infoName object:self userInfo:@{@"info":@"change background color"}];

postNotificationName: 与上面观察者的消息专有名字一样
object: 发送消息的对象
userInfo: 消息包含的内容，将会传到消息接收者处

最后要记得
remove 掉注册者

NSNotificationCenter

  • 每一个 App 运行的时候都有的一个单例对象
  • 通知中心同步转发通知给观察者，就是说 postNotification: 方法直到接收并处理完通知才返回值

建立硬盘分区

By Wen Yongyang

发表于 2015-11-10

建立硬盘分区)

  • genw划分多个逻辑区块，区块间的访问相互独立
  • 可从逻辑上隔离数据
  • 每个分区使用前需格式化为文件系统

分区表

  • 分区信息存放地点
  • 主流模式：
    • 传统：Master Boot Record (MBR)
    • 新型：GUID Partition Table (GPT)

Master Boot Record

  • 最多支持4个分区
  • 利用扩展分区和逻辑分区绕过上述限制
  • 分区类型
    • 主分区 (Primary)
    • 扩展分区 (Extended)
      • 逻辑分区 (Logical)
      • 扩展分区 可看作 逻辑分区 的容器
      • 扩展分区包含的逻辑分区没有限制
  • 通常习惯
    • 创建主分区 sda1 到 sda3，建立一个扩展分区 sda4
    • sda4 中包含 sda5，sda6 等逻辑分区

GPT

  • 只有一种分区类型：主分区

必须使用 MBR 情况

  • 使用 GRUB legacy 作为 bootloader
  • 使用传统的 BIOS，且双启动中包含 Windows(32/64)

必须使用 GPT

  • 使用 UEFI 而不是 BIOS，且双启动中包含 Windows 64

分区方案

  • 单 root 分区
  • 多分区
    • 将某个路径挂载为独立分区，使其拥有不同的文件系统和挂在参数，可被多个操作系统共享

挂载点

  • / (根分区)
    • 目录树顶层
    • 主文件系统挂载和其它文件系统挂载的地方
    • 其中的内容应满足于：
      • 启动
      • 恢复
      • 修复系统
    • 其下特定目录不能作为独立分区
  • /boot
    • 包含
      • 内核
      • ramdisk 镜像
      • bootloader 配置文件
      • bootloader stage
    • 内核在使用用户态程序前使用的其它数据
    • 日常系统运行并不需要，只在启动，内核升级（重建 initial ramdisk）时用到
  • /home
    • 包含
      • 用户定义的配置文件、缓存、应用程序数据、媒体文件
      • 可在不修改 /home 内容情况下重装 Arch，删除其它顶级目录，执行 pacstrap
  • /var
    • 存储：
      • 变量数据，如 spool 目录和文件
      • 管理和登录数据
      • pacman 的缓存
      • ABS 树。。。。
    • 常用做 缓存或日志记录，读写频繁
    • 独立作用：避免由于大量日志写入造成磁盘空间耗尽
  • /tmp
    • 默认情况下，已经是独立分区
  • Swap
    • 虚拟内存空间

分区大小设置

  • /boot
    • 200MB
    • 实际需要约 100MB
  • /
    • 15-20GB
    • 传统上包括 /usr 目录
  • /var
    • 8-12GB
  • /home
    • 不定
    • 在桌面系统中，通常是最大的文件系统
  • Swap
    • 不定，通常是物理内存的2倍

分区工具

  • fdisk Linux 自带命令行分区工具

iOS多线程

By Wen Yongyang

发表于 2015-11-08

###多线程编程层次

  • Thread
    • 层次最低
    • 线程的管理，线程锁设置与释放等需要开发者负责
    • 使用NSThread来创建和管理
  • Cocoa Operations
    • 使用NSOperation类来管理线程
    • 已封装好关于线程的管理，锁等问题
    • NSOperation是抽象基类
    • iOS提供的默认实现：NSInvocationOperation，NSBlockOperation
    • 实现多线程技术方法：Operation Queue 操作队列
      • 开发者任务：定义需要执行的任务，将其加入到操作队列中
      • 操作队列负责调度和执行任务

  • Grand Central Dispatch (GCD)

    • 可将应用程序需要执行的工作拆分为可分散在多个线程和多个CPU上更小的块
    • 大部分API使用Block来访问
    • 异步启动任务的技术之一，GCD负责线程管理的代码，并将线程管理移到更低端的系统层面
    • 开发者的任务：定义需要执行的任务，并把任务添加到合适的派遣队列 Dispatch Queue中

  • iOS基础框架中所有Objective-C的基础类，如NSString，NSArray等被视为线程安全

  • UIKit框架包含的用于构建GUI的类，如UIApplication，UIView及其所有子类是线程不安全的
    • 处理UIKit对象的所有方法都放在主线程中

###GCD提供的默认派遣队列

  • Private Dispatch Queue 顺序队列
    • 通常用于同步访问特定的资源
    • 一个队列中按照队列的性质，一次执行一个任务
    • 可创建任意多个顺序队列，每一个队列同步地独立于其他队列而运行
  • Global Dispatch Queue 并发队列，全局派遣队列
    • 同步地执行一个或多个任务，而启动顺序按照队列性质
    • 管理多个不同的线程，并发的任务在管理的不同线程中执行
    • 某个时间点队列里执行的任务数是可变的
    • 开发者不能创建自己的并发队列，只可使用iOS提供的3个全局并发队列
  • 主队列
    • 在应用程序的主线程执行的队列，与应用程序的运行循环交错执行

###派遣队列的创建与执行

  • 顺序队列

    • 异步执行

      dispatch_queue_t queue;
queue = dispatch_queue_create("Queue name",NULL);//由于这是C语言，使用NULL
//异步执行
dispatch_async(queue,^{
    [NSThread sleepForTimeInterval:5];
    NSLog(@"Running in the queue");
});

    • 同步执行

      dispatch_queue_t queue;
queue = dispatch_queue_create("Queue name",NULL);
//同步执行
dispatch_sync(queue,^{
    [NSThread sleepForTimeInterval:5];
    NSLog(@"Running in the queue");
});
      • 用于防止资源访问冲突

  • 并发队列

    dispatch_queue_t queue;
queue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT,0);
dispatch_async(queue,^{
    [NSThread sleepForTimeInterval:5];
    NSLog(@"First task in the queue");
});
dispatch_async(queue,^{
    [NSThread sleepForTimeInterval:5];
    NSLog(@"Second task in the queue");
});
dispatch_async(queue,^{
    [NSThread sleepForTimeInterval:5];
    NSLog(@"Third task in the queue");
});
    • 并发队列有3种优先级
      • 高 DISPATCH_QUEUE_PRIORITY_HIGH
      • 低 DISPATCH_QUEUE_PRIORITY_LOW
      • 普通 DISPATCH_QUEUE_PRIORITY_DEFAULT
    • 第二个参数用来扩展使用，暂且为0

  • 主线程队列

    dipatch_queue_t queue;
queue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT,0);
dispatch_async(queue,^{
    //做一些运算

    //然后这里才是主线程调用
    dispatch_async(dispatch_get_main_queue(),^{
        aLabel.text = @"Processed by the main queue";
    });
});

###添加任务到派遣队列

  • 同步
    • 同步的添加方法与异步对应，但添加完后需等待代码执行完才返回

  • 异步

    • dispatch_async
      • void dispatch_async(dispatch_queue_t queue, dispatch_block_t block);
      • arg1: 代码块要添加到的队列
      • arg2: 要执行的代码块
      • 此方法添加完代码块后立即返回，不会等待代码执行

    • dispatch_async_f

      void dispatch_async_f(
    dispatch_queue_t queue,
    void *context,
    dispatch_function_t work
);
      • arg3: 需要执行的函数
      • arg2: 函数的参数

###执行 回调函数

  1. 在创建任务的同时指定任务完成时的处理方法
  2. 在队列中嵌套队列来实现，如主线程的创建与执行例子

###派遣组技术

  • 允许阻止线程直到一个或者多个任务执行完成

  • 让一个或者多个队列等待执行完毕后再返回主线程

    dispatch_queue_t queue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT,0);
dispatch_group_t group = dispatch_group_creat();

dispatch_group_async(group,queue,^{
    //执行异步操作
});

//一些其他处理

//在派遣组中等待，阻止当前的线程继续执行
dispatch_group_wait(group, DISPATCH_TIME_FOREVER);

###代码块Block用途

  • 枚举
  • 视图动画效果，如animateWithDuration:animations:
  • 排序
  • 通知：当某种情形发生时执行指定代码块
  • 错误处理：当错误发生时，执行指定代码块
  • 事件完成处理器

  • 例子：对字符串数组进行排列

    NSArray *strArray = [NSArray arrayWithObjects:@"...",@"...",...,nil];

//获取当前语言环境
NSLocale *currentLocale = [NSLocale currentLocale];
NSArray *diacriticInsensitiveSortArray = [strArray soredArrayUsingComparator:^(id str1, id str2){
    NSRange str1Range = NSMakeRange(0,[str1 length]);
    NSComparisonResult comparisonResult = [str1 compare: str2 options: NSDiacriticInsensitiveSearch range: str1Range locale: currentLocale];
    return comparisonResult;
}];

NSLog(@"%@",diacriticInsensitiveSortArray);
    • currentLocale获取当前语言环境，用来确认排序的规则（因语言环境不同而不同）
    • diacriticInsensitiveSortArray 存储排列结果
    • 在代码块中，没有声明返回类型，因为iOS能根据代码块内部返回值类型推断

###异步编程方法

  • 使用队列
  • 通过performSelector
  • 通过NSTimer

###通过performSelector

  • - (void) performSelector: (SEL)aSelector withObject: (id)anArgument afterDelay: (NSTimerInterval)delay;
    • arg3: 指定方法在多长时间以后开始执行
    • arg1: 指定时间到了的时候执行的方法
    • arg2: 传递给arg1的参数
    • 该方法调用后立即返回，时间到了会执行指定方法
    • 如：[self.label performSelector:@selector(setText:) withObject:msg afterDelay:5.0];
  • - (void) performSelectorOnMainThread: (SEL)aSelector withObject: (id)arg waitUntilDone: (BOOL)wait;
    • arg3: 是否阻塞队列来等待方法执行完毕

###通过NSTimer

  • + (NSTimer *) scheduledTimerWithTimeInterval: (NSTimeInterval)ti target: (id)aTarget selector: (SEL)aSelector userInfo: (id)userInfo repeats: (BOOL)yesOrNo
    • (NSTimeInterval)ti 时间间隔以秒为单位
    • target: (id)aTarget 被执行方法所处的类
    • userInfo: (id)userInfo 由NSTimer持有的信息，暂且为nil
    • repeats: (BOOL)yesOrNo 是否重复调用
