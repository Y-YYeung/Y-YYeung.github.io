The Role of View Controllers 视图管理器的作用

By Wen Yongyang

發表於 2015-11-29

文章目錄

View Controller 视图控制器

View Controllers 是应用内部结构的基础。每一个应用至少有一个 View Controllers，但大部分的应用会有多个。每一个 View Controllers 管理应用中部分的用户交互界面以及用户交互界面与低层数据的交互。同时，使用 View Controllers 也有利于应用中不同部分界面之间的相互过渡。

View Controllers 在应用中扮演着非常重要的角色，因此，View Controllers 几乎是你工作的重心。UIViewController 类定义了多个方法(methods) 和属性(properties) 来管理视图(views)，事件处理(handling events)，视图控制器间的过渡切换(transitioning from one view controller to another)，协调处理应用的各部分。开发者应该继承 UIViewController （或其子类），并在这些类中添加自定义的代码来实现应用的功能。

其中，有两种类型的 View Controllers:

  • Content view controllers（内容类视图控制器），管理应用內零散的内容，这是开发者主要会创建的 view controller
  • Container view controller（容器类视图管理器），从其他 view controllers（child view controllers）收集信息，并以导航的方式或以其他方式将其他的 view controllers 展现出来。

大多数应用都是上述两类 view controller 的混合体。

##View Management 视图管理

View controller 最重要的功能就是管理视图之间的层次关系。每一个 view controller 都有唯一的一个 root view（根视图），该 root view 包含了 view controller 的所有内容。在这个 root view 上，开发者添加想要展现的内容。Figure 1-1 描述了 view controller 和它的 views 之间的内部关系。View controller 总是有对其 root view 的引用，而每一个 view 都有对其 subviews 的强引用。

Figure 1-1Relationship between a view controller and its views

NOTE
使用 outlet 访问其他在 view controller 的 view 层次结构中的 view（如子控件）是一种很通用的做法。由于 view controller 管理其所有 view 的内容，因此，outlet 为开发者存储了对 views 进行访问时需要的引用。当 views 从 storyboard 加载时，outlet 会自动与 view 的实例关联起来。

Content view controller 只管理自身包含的 views。Container view controller 管理自身包含的 views 以及它的一个或多个 child view controllers 中的 root views。Container 并不管理其子控制器的 content。它只管理 root view，根据 container 的设计来计算大小位置并进行放置。Figure 1-2 描述了 split view controller 和它的子控制器。Split view controller 管理其 child views 的总体大小和位置，但 child view controller 才是真正管理那些 views 的控制器。

Figure 1-2 View controllers can manage content from other view controllers

##Data Marshaling 数据整理

View controller 扮演一个中间人(intermediary)的角色，协调它管理的 views 和应用的数据。UIViewController 类中的方法和属性能够让开发者管理应用的界面呈现效果。当继承 UIViewController 后，开发者可以向这个子类添加管理数据所需要的变量，向这个子类添加自定义变量时，会创建如 Figure 1-3 所示的关联，在这里，view controller 对数据和呈现数据的 views 都会有 references（引用）。在 view controller 和 view 之间来回传递数据是开发者的职责。

Figure 1-3A view controller mediates between data objects and views

开发者应明确清晰地对 view controllers 和数据对象有一种清晰的任务分工。大多数确保数据结构统一正确的逻辑工作应该被包含在数据对象自身中(即大多数数据处理逻辑代码应该写在数据类中)。而 view controller 则可以对从 views 来的数据进行验证并将数据打包成数据对象需要的特定格式，但要记住的是，要尽可能降低 view controller 在管理数据的任务。

使用 UIDocument 对象是独立于 view controllers 外管理数据的一种方法。Document 对象是一个能如何进行持久化存储的 controller 对象。当继承这个类时，你可以在其中添加你需要的方法或逻辑来读取数据或将数据传递到其他控制器或其他部分。View controller 可以存储它接收到的数据的一份副本，以此来方便更新视图，但 document 对象仍然拥有着真正的数据。

##User Interactions 用户交互

View controllers 是响应事件的对象，并具有处理从 responder chain（响应链） 传递下来的事件的能力。尽管如此，view controllers 很少直接处理触摸事件。相反，views 通常会处理它们自身的触摸事件，并将结果返回到一个 associated delegate（关联的代理方法）或一个 target object（目标对象，通常为当前的 view controller）。因此，大多数在 view controller 中的事件会通过 delegate methods（代理方法）或 action methods（动作方法）来处理。

More Info: Event Handling User Interactions And Event Handling Guide for iOS

##Resource Management 资源管理

View controller 默认对所有其创建的 views 和对象负责。UIViewController 类自动处理大多数层面的视图管理。例如：UIKit 自动释放那些不再需要的与视图关联的资源。在开发者自己创建的 UIViewController 子类中，开发者承担管理所有自己显式创建对象的责任。

当可用内存空间不足时，UIKit 请求应用释放那些不再需要的资源。其中一个请求方法就是系统调用 view controllers 中的 didReceiveMemoryWarning 方法。开发者使用这个方法，在里面编写代码，移除那些不再需要或者可以稍后重新创建的资源的引用。例如：开发者可以使用这个方法来移除缓存数据。当内存空间不足时，开发者应释放尽可能多的内存空间。而消耗太多资源的应用可能会被系统终止并完全移出内存来恢复内存空间。

##Adaptivity 适配

View controller 负责呈现 views 和调整呈现方式来匹配低层环境的工作。每一个 iOS 应用都应该能运行在 iPad 和不同尺寸的 iPhone 上。简单的做法是使用单一的 view controller 来调整 views 来适配不同的空间变化，而不是为每一种设备编写不同的 view controllers 和 view 层次结构。

在 iOS 中，view controller 需要处理 coarse-grained 变化和 fine-grained 变化。Coarse-grained 变化是当一个 view controller 的 trait 发生变化时产生。 Traits 是描述整体环境的属性，例如 display scale（显示比例）。其中最重要的两个 traits 是 view controller 的水平和垂直方向的 size classes （尺寸比列），这两个 traits 表示了 view controller 在给定的维度中有多大的空间。开发者可以使用 size class 变化来改变 views 的布局方式，如 Figure 1-4 所示。当水平 size class 为 regular 时，view controller 利用额外的水平空间来布局它的内容。当水平 size class 为 compact 时，view controller 垂直布局它的内容。

Figure 1-4 Adapting views to size class changes

在一个给定的 size class 中，给了 fine-grained 大小发生变化的可能。当用户将 iPhone 从 portrait（竖屏）转换为 landscape（横屏）时，size class 也许不会发生变化，但屏幕的尺寸却会发生改变。当开发者使用 Auto Layout （自动布局）时，UIKit 会自动调整 views 的大小和位置来适配新的尺寸。View controllers 可根据需要添加额外的调整。
